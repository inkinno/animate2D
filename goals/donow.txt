0. 아직 읽지않았다면 4개의 문서 먼저 읽어보기를 바람. 그 goals들로부터 현재 해야할일들에대한 예시. 즉 세부 내용들도 예시에 불과하며. goals를 읽고 너가 판단한 내용을 다음처럼 순서를 나누어 계획을 세우고. 다른 ai가 와서 해도 context를 잃지 않도록 md파일화 하여 새로운 파일로 만들어서 root폴더에 넣어라. 다음예시처럼 파일로 어떤것들을 해야할지 만들어서 넣고. 해당 하는 내용의 코드를 바로 짜는것이 아닌 세부 계획을 포함한 블루프린트 확대 버전을 만들도록.

1. 전역 파일 시스템 아키텍처 (File Tree Map)
가장 먼저 물리적인 파일 디렉토리 구조를 설계해야 합니다. 시스템은 완벽히 디커플링(Decoupling)된 3대 요소로 구성되므로, 코드가 서로 침범하지 않도록 경계를 긋는 작업입니다.

설계 목표: Core Engine, Player, Maker가 각각 독립적으로 빌드되고 테스트될 수 있는 폴더 구조를 확립합니다.

포함되어야 할 내용:

packages/core-engine: 수학 행렬 연산 및 델타 타임 루프가 들어갈 순수 로직 디렉토리.

packages/player: WebGL 캔버스 렌더러와 로컬 스토리지 기반 상태 저장 모듈이 위치할 디렉토리.

packages/maker: 4분할 레이아웃 GUI와 리액트(혹은 뷰) 컴포넌트가 들어갈 앱 디렉토리.

packages/shared: 공통으로 사용할 유틸리티와 타입(Type) 정의 폴더.

2. 마스터 데이터 스키마 맵 (JSON Interface Map)
코딩 전 가장 중요한 '계약서'입니다. 엔진, 플레이어, 메이커가 상호 교환할 단일 파일 데이터의 표준 규격 사양을 먼저 확정해야 합니다.

설계 목표: 뼈대(Bone), 파츠(Parts), 타임라인 키프레임, 이징 수치, 동적 슬롯 등이 JSON 형태로 어떻게 표기될지 AI와 인터페이스를 정의합니다.

포함되어야 할 내용:

루트 노드와 계층형 자식 노드의 데이터 상속 구조 표현 방식.

아핀 변환(x, y, rotation, scaleX, scaleY) 데이터의 소수점 규격.

메이커에서 결정한 이벤트 트리거 및 레이어 스왑(Z-Index) 데이터의 삽입 위치.

3. 상태 및 데이터 흐름도 (State & Data Flow Map)
데이터가 각 모듈 사이를 어떻게 흘러가고 언제 저장되는지 라이프사이클을 매핑합니다.

설계 목표: 유저의 행동에 따라 시스템이 어떻게 반응하고 메모리를 관리하는지 규정합니다.

포함되어야 할 내용:


Player 측: 비동기 다운로드 큐의 흐름과, 브라우저 종료 시 진행 데이터를 실시간(Real-time) 비동기 스냅샷으로 저장하는 로직의 순서도.

Engine 측: 가비지 컬렉션을 막기 위한 메모리 풀링(Object Pooling) 초기화 및 프레임당 행렬 연산 순서도.

4. 메이커 UI 컴포넌트 맵 (Maker Component Architecture)
메이커는 매우 복잡한 상태 관리가 필요하므로, 화면을 어떻게 쪼개어 개발할지 컴포넌트 트리를 그립니다.


설계 목표: 4분할 영역으로 고정 배치 구조를 가지는 메이커의 프론트엔드 뼈대를 시각화합니다.

포함되어야 할 내용:

상태 관리(Store) 구조: 타임라인 데이터, 선택된 파츠 상태, 현재 프레임 위치 등을 관리할 전역 상태 맵.

좌측 탐색기, 중앙 뷰포트, 우측 속성 데이터 편집 패널, 하단 타임라인 시퀀서의 부모-자식 컴포넌트 관계도.
