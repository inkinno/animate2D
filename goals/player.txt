[Document 2] 플레이어 기획서 : 클라이언트단 엔진 구동 및 렌더링 명세서 (Player Runtime Specification)
1. 플레이어 실행 매커니즘 및 배포 인터페이스 (Execution & Interface)
플레이어는 독립된 HTML5 Canvas 객체 단일 레이어로 구동되며, 외부 리소스 인터벤션 없이 완전히 독립된 샌드박스(Sandbox) 형태로 동작한다.

배포 인터페이스는 웹 프레임워크와의 결합성을 유연하게 유지하기 위해 완전한 자바스크립트 API 인젝션(JavaScript API Injection) 방식을 채택한다. 특정 HTML 컨테이너(div) 엘리먼트의 ID를 주입받아 그 내부에 캔버스를 동적 생성하는 구조로 설계한다.

런타임 구동을 위한 메인 진입 함수는 단일 기둥 데이터인 JSON 스키마 주소를 매개변수로 받아 비동기적으로 플레이어 생성 루틴을 개시한다.

2. 가용 영역 계산 기반 반응형 캔버스 아키텍처 (Dynamic Layout)
플레이어는 스마트폰 및 디바이스 스크린 전체 크기를 기준으로 캔버스를 생성하지 않는다. 스크린 내부에서 상단 로고, 텍스트 대화창, 하단 광고 영역(Ads Area) 등 캔버스 외부의 필수 구성요소들이 차지하는 레이아웃 스페이스를 최우선으로 선행 계산한다.

외부 고정 레이아웃 요소를 전부 제외하고 남은 '실제 가용 화면 영역(Actual Available Viewport)'을 확보한 뒤, 가로와 세로 중 가장 작은(Minimum) 축의 절대 픽셀 길이를 도출한다.

도출된 최소 축 길이를 기준으로 메이커가 규정한 고정 종횡비(예: 1:1 정사각 또는 2:3 세로 비율) 모델을 적용하여 캔버스 외각 크기를 최종 확정한다. 이를 통해 외부 UI 엘리먼트와 플레이어 캔버스가 겹치거나 화면 밖으로 넘치는 오버플로우(Overflow) 현상을 원천 차단한다.

종횡비 캔버스의 화면 배치 정렬 정책은 상단 정렬(Top Alignment)을 기본값으로 강제한다. 상단에는 캔버스 영역이 시각화되고, 캔버스 바로 아래쪽 빈 공간에는 선택지 및 대화창 텍스트 레이아웃이 배치되며, 최하단(Bottom)에 광고 레이어가 결합하는 수직 적층형 인터페이스 구조를 형성한다.

3. 동적 에셋 프리로딩 큐 및 다운로드 인터럽트 논리 (Asset Streaming Engine)
플레이어는 무조건 선 로딩 후 플레이(Preload Before Play) 메커니즘을 엄격히 준수한다. 실행 직후 캔버스 내부에 움직이는 인디케이터(Loading Indicator UI)를 즉시 송출하여 유저 진입 경험을 보호한다.

다운로드 파이프라인은 메이커에서 지정한 타임라인 씬(Scene) 배치 순서 규칙을 기반으로 순차적 자동 다운로드 큐(Asynchronous Download Queue)를 구동한다. 기본 세팅은 총 에셋 크기에 따라 100MB 이상일 경우 씬 단위 순차 실시간 다운로드를 수행하고, 100MB 미만일 경우 전체 통 다운로드(Bulk Download) 후 재생을 시작한다.

씬 단위 순차 다운로드 중 유저 상호작용에 의해 분기형 씬 전환(예: 씬 1 종료 후 씬 2가 아닌 씬 5 선택)이 발생할 경우 다음의 네트워크 인터럽트 연산을 실행한다.

1단계: 네트워크 중복 체크 알고리즘을 실행하여 현재 큐에 대기 중이거나 이미 VRAM에 적재 완료된 에셋 중 씬 5에 필요한 리소스가 있는지 검증하고, 중복 에셋은 재활용 프로세스로 즉시 바이패스(Bypass)한다.

2단계: 씬 5에 없는 리소스 중 현재 백그라운드에서 전송 중이던 다른 분기 씬(예: 씬 3, 씬 4)의 HTTP Fetch/XHR 요청 컬렉션을 즉시 중단(Abort/Cancel) 시킨다. 단, 현재 시점에 다운로드가 거의 마무리 단계에 진입한 에셋 조각은 네트워크 오버헤드를 줄이기 위해 마무리를 허용한다.

3단계: 다운로드가 취소된 씬들에 대한 다운로드 명령을 큐에서 영구 파기하고, 씬 5에 필요한 미확보 에셋 리스트를 최상위 우선순위(High-Priority)로 큐의 맨 앞에 동적 배치한다.

4단계: 만약 씬 5의 첫 프레임이 실행되는 시점까지 필수 스프라이트가 미 다운로드 상태일 경우, 엔진은 플레이어를 즉시 대기(Suspended) 상태로 전환하고 전용 로딩 인디케이터 화면을 화면에 안전하게 송출하여 타임라인 싱크가 깨지는 현상을 방지한다.

4. 타임라인 제어 알고리즘 및 양방향 스크롤 매핑 (Timeline Controller)
플레이어는 일반적인 재생 환경에서는 타임라인 스크러버 UI를 노출하지 않는 무형의 상태를 기본값으로 취한다. 단, 메이커에서 특정 씬에 대해 명시적으로 앞뒤 탐색 권한을 허용했을 경우에 한하여 제어 논리를 활성화한다.

웹툰 모드 구동을 위한 스크롤 연동 시, 시스템 메모리 보존 및 안정적인 CPU 점유율 유지를 위해 메이커 단에서 '양방향 스크롤 매핑'과 '단방향 트리거' 모드를 선택 제어할 수 있는 세그먼트 스위칭 논리를 내장한다.

양방향 스크롤 매핑 모드: 메모리 파편화 가드레일 범위 내에서 독자의 마우스 휠 및 터치 스크롤 델타값을 실시간으로 양방향 변환 행렬 스택(Matrix Stack)에 매핑한다. 역방향 스크롤 이벤트 수신 시, 타임라인 경과 시간(ms)을 마이너스 역산하여 부모-자식 본들의 변형 상태(SRT)를 키프레임 이전 지점으로 실시간 복원 렌더링한다.

단방향 트리거 모드: 스크롤이 브라우저 뷰포트의 특정 임계점(Trigger Boundary)을 터치하는 순간, 플레이어는 해당 세그먼트 애니메이션을 결정론적 정방향으로 오직 1회만 구동하고 타임라인 연산을 완벽히 정지(Freeze)시킨다.

5. 실시간 상태 저장 메커니즘 및 런타임 비정상 종료 대책 (State Persistence)
플레이어는 앱 환경이 아닌 브라우저 환경에서 동작하므로 유저의 이탈(뒤로가기, 새로고침, 탭 닫기)에 실시간 대응하기 위해 '영속적 상태 저장 모듈(State Persistence Module)'을 런타임 코어에 내장한다.

실시간 진행 상황 스냅샷: 플레이어는 타임라인 재생 중 유저가 어느 세그먼트 씬의 몇 밀리초(ms) 지점까지 감상했는지, 상호작용에서 어떤 상태 분기값(State Flag)을 선택했는지에 대한 핵심 데이터를 브라우저의 IndexedDB 또는 LocalStorage에 실시간(Real-time) 비동기 스냅샷으로 저장한다.

컨텍스트 체크 및 영속 로딩: 비정상적인 종료 이후 웹 앱이 재구동되면, 플레이어 진입 함수는 서버나 초기 JSON의 기본값을 무조건 신뢰하지 않고 클라이언트 로컬 저장소를 선행 검사한다. 저장된 유저 세이브 데이터가 포착되면 해당 최종 체크포인트 지점을 기점으로 연산 장치를 정렬하고, 필요한 파츠만 타깃 다운로드 큐에 후입선출(LIFO) 방식으로 우선 주입하여 즉각적인 화면 복구를 수행한다.

범용 세이브/로드(Save/Load) 커맨드: 특정 에피소드 방식이나 게임형 시나리오 연출을 지원하기 위해, 플레이어 코어 API는 현재 런타임 가상 메모리 매트릭스 전체 상태를 직렬화(Serialization)하여 저장 및 호출할 수 있는 공용 메서드를 개방한다.

초기화 인터랙션 강제: 플레이어 외부의 UI 영역 혹은 시스템 특정 영역에 상시 호출이 가능한 '처음으로 돌아가는 버튼(Reset Trigger)' 커맨드를 강제 매핑한다. 유저가 리셋을 트리거할 경우 로컬에 실시간 축적되던 진행 데이터 스택을 즉시 폐기하고, 타임라인 포인터를 0ms로 회귀시켜 전체 씬 데이터와 다운로드 큐를 마스터 리셋(Master Reset) 상태로 초기화한다.

6. 인터랙션 상태의 정적 반복 루프 자원 배정 (Runtime Idle State)
실시간 타임라인 시퀀스의 재생이 종료되거나 유저의 버튼 선택 입력을 기다리는 '상호작용 씬' 상태에 돌입하더라도 렌더링 루프(requestAnimationFrame)를 물리적으로 완전 정지(Pause)시키지 않는다. 캐릭터와 오브젝트가 캔버스 내부에서 시각적으로 생동감을 유지할 수 있도록 정적 대기 상태(Idle State) 모드로 전환한다.

정적 대기 상태 모드 진입 시 플레이어는 메이커에서 사전 지정한 미세한 숨쉬기(Breathing Variance), 안구 깜빡임(Blink Tick), 미세 대기 바운스 동작에 해당하는 최소한의 전력 루프(Static Animation Loop) 레이어만 부분적으로 구동한다.

메인 타임라인 연산 장치는 대기 상태 동안 일시 휴면 구조로 전환되어 CPU 메모리 소비량을 최소화하며, 오직 프리로딩된 정적 루프 키프레임의 보간값만 GPU 매트릭스 버퍼에 지속 전달하여 프레임 레이트 60fps 유지와 기기 전력 효율성을 동시에 확보한다.

사용자가 상호작용 요소(버튼, 특정 좌표 터치)를 최종 클릭하여 신호가 입력되면 대기 상태 루프는 즉시 컨텍스트를 비우고(Clear), 다음 타임라인의 첫 번째 프레임 행렬 연산부로 제어권을 즉각 전이시킨다.

7. 라이프사이클 및 연산 종료 명세 (Lifecycle Termination)
애니메이션의 최종 씬이 종료되고 연출가가 설정한 엔딩 크레딧 혹은 엔딩 정적 화면(Static Ending Layer)에 도달하면, 플레이어 내부의 모든 실시간 타임라인 역학 계산 및 이징 보간 루프를 완벽히 셧다운(Shutdown)한다.

연산이 종료된 캔버스는 다시 재생 버튼이 트리거되거나 외부 JS API를 통해 명시적인 활성화 명령이 전달되기 전까지 '완전 정지 상태(Absolute Dormant Status)'를 유지한다.

정지 상태에서는 유저가 엔딩 화면 내 정적 버튼을 누르는 물리적인 마우스 포인트 인터랙션(Button Hover/Click) 연산만을 제외한 모든 리얼타임 본 행렬 계산 루틴을 메모리에서 해제하여 메모리 누수와 불필요한 프레임 갱신을 완전히 종결한다.
